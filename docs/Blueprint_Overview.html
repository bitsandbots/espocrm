EspoCRM Accounting Integrations

Operational blueprint for the CoreConduit EspoCRM deployment with bidirectional QuickBooks Online and Xero sync. Covers architecture, data flows, module structure, setup, and open gaps.

1. Project Overview

This deployment extends the open-source EspoCRM platform with two custom accounting integration modules: QuickBooks Online (QBO) and Xero. The goal is to eliminate duplicate data entry between CRM operations (contacts, accounts, deals) and accounting (customers, invoices, payments).

CRM — EspoCRM

  • Source of truth for Accounts, Contacts, Leads, Opportunities
  • Custom Invoice entity
  • Activity stream, email, tasks, meetings
  • Role-based access control

Accounting — Dual Support

  • QuickBooks Online (Intuit)
  • Xero (cloud-first)
  • Receive Customers / Contacts
  • Receive Invoices; pull Payments

Sync Strategy

  • Hook-on-save for real-time push
  • Nightly scheduled pull for changes
  • Last-modified-wins conflict resolution
  • Loop guard via skipQuickBooksSync / skipXeroSync

Runtime

  • PHP 8.3, Slim 4 framework
  • MySQL 8 / MariaDB / PostgreSQL
  • JavaScript SPA (AMD modules)
  • Background jobs via cron

5. QuickBooks & Xero OAuth2 Flows

QuickBooks OAuth2 Flow

  1. Admin opens Administration → Integrations → QuickBooks and saves clientId + clientSecret (obtained from developer.intuit.com).
  2. Admin navigates to the QB authorization URL:
    https://appcenter.intuit.com/connect/oauth2?client_id=<id>&scope=com.intuit.quickbooks.accounting&redirect_uri=<siteUrl>?entryPoint=QuickBooksOauthCallback&response_type=code&state=<token>
  3. User logs into QuickBooks and grants access.
  4. QB redirects to ?entryPoint=QuickBooksOauthCallback&code=…&realmId=…&state=…
  5. The entry point exchanges the code for tokens (HTTP Basic auth with clientId:clientSecret), then writes accessToken, refreshToken, realmId, and connectedAt to the Integration entity.
  6. Tokens auto-refresh (30-second expiry margin) on every API call in QuickBooksService.

QB Redirect URI to register in QB Developer Portal:

{siteUrl}?entryPoint=QuickBooksOauthCallback

Xero OAuth2 Flow

  1. Admin opens Administration → Integrations → Xero and saves clientId + clientSecret (obtained from developer.xero.com).
  2. Admin navigates to the Xero authorization URL:
    https://login.xero.com/identity/connect/authorize?client_id=<id>&response_type=code&scope=accounting+offline_access&redirect_uri=<siteUrl>?entryPoint=XeroOauthCallback&state=<token>
  3. User logs into Xero and grants access.
  4. Xero redirects to ?entryPoint=XeroOauthCallback&code=…&state=…
  5. The entry point exchanges the code for tokens (HTTP POST with clientId:clientSecret in body), then writes accessToken, refreshToken, tenantId, and connectedAt to the Integration entity.
  6. Tokens auto-refresh on every API call in XeroService.

Xero Redirect URI to register in Xero Developer Portal (HTTPS required):

{siteUrl}?entryPoint=XeroOauthCallback

6. HTTPS & TLS Configuration

Xero OAuth requires HTTPS. The deployment uses mkcert to issue locally-signed TLS certificates on cake.local port 8443.

Setup mkcert (one-time)

brew install mkcert nss  # or apt-get on Linux
mkcert -install
cd /home/coreconduit/espocrm
mkcert -cert-file=cake.local.pem -key-file=cake.local-key.pem cake.local

nginx TLS block

server {
  listen 8443 ssl http2;
  server_name cake.local;

  ssl_certificate /home/coreconduit/espocrm/cake.local.pem;
  ssl_certificate_key /home/coreconduit/espocrm/cake.local-key.pem;

  # ... rest of config
}

For local testing: Add 127.0.0.1 cake.local to /etc/hosts, then register both HTTP and HTTPS URIs in OAuth app settings:

  • QuickBooks: https://cake.local:8443?entryPoint=QuickBooksOauthCallback
  • Xero: https://cake.local:8443?entryPoint=XeroOauthCallback

7. Setup & Activation

Prerequisites

  • PHP 8.3+, MySQL 8+ (or MariaDB/PostgreSQL), web server (nginx/Apache)
  • Composer 2.x
  • HTTPS configured (mkcert for local dev, production CA for staging/prod)
  • QuickBooks Online account + developer app at developer.intuit.com (optional)
  • Xero account + developer app at developer.xero.com (optional)
  • Cron configured to run EspoCRM's cron.php every minute

Initial Setup

cd /home/coreconduit/espocrm

# Install all dependencies (including dev tools)
composer install

# Transpile JS (QB and Xero AMD modules)
node js/transpile.js

# Register custom modules, rebuild metadata + DB schema
php rebuild.php

# Verify scheduled jobs are registered
php command.php run-job --help

QuickBooks Integration Activation

  1. Create a QB developer app. Set the redirect URI to {siteUrl}?entryPoint=QuickBooksOauthCallback. Note the clientId and clientSecret.
  2. In EspoCRM: Admin → Integrations → QuickBooks. Enable it, enter credentials, save.
  3. Manually open the QB OAuth URL (see §5) in a browser. Authorize. The callback stores tokens automatically.
  4. In Admin → Scheduled Jobs, add two jobs: Espo\Modules\QuickBooks\Jobs\SyncFromQuickBooks (daily) and Espo\Modules\QuickBooks\Jobs\ReconcileQuickBooks (daily, runs after Sync).
  5. Run php rebuild.php once more to clear caches.
  6. Test: edit an Account in EspoCRM — it should appear as a QB Customer within seconds.

Xero Integration Activation

  1. Create a Xero developer app at developer.xero.com. Set the redirect URI to {siteUrl}?entryPoint=XeroOauthCallback. Note the clientId and clientSecret.
  2. Ensure HTTPS is configured (see §6).
  3. In EspoCRM: Admin → Integrations → Xero. Enable it, enter credentials, save.
  4. Manually open the Xero OAuth URL (see §5) in a browser. Authorize. The callback stores tokens automatically.
  5. In Admin → Scheduled Jobs, add two jobs: Espo\Modules\Xero\Jobs\SyncFromXero (daily) and Espo\Modules\Xero\Jobs\ReconcileXero (daily, runs after Sync).
  6. Run php rebuild.php once more to clear caches.
  7. Test: edit an Account in EspoCRM — it should appear as a Xero Contact (IsCustomer) within seconds.

9. Known Gaps & Limitations

  • No Invoice CRUD controller: Invoices are created/updated only via sync jobs, not through the REST API. Manual Invoice creation in EspoCRM is not supported.
  • No disconnect endpoint: Once connected, QB and Xero authorizations cannot be revoked from within EspoCRM. Must be done in QB/Xero web UIs.
  • No webhook support: Sync is polling-based only (nightly). Real-time bidirectional sync requires Xero/QB webhooks (future work).
  • No sync audit log: Sync operations are logged to Monolog but not exposed in the admin UI. Debugging sync failures requires log file access.
  • Opportunity & Lead mapping: QB Estimates and Xero Quotes are not mapped; Lead syncing is not implemented.
